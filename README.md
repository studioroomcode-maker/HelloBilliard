# Hello Billiard — 당구 AI 코치

4구 경로 시뮬레이터 + 3구(쓰리쿠션) 시스템 계산기. 공을 배치하거나 **당구대 사진을 찍으면** 물리 시뮬레이션으로 득점 경로를 성공률과 함께 제안합니다.

**라이브(PWA)**: https://hellobilliard.studioroomkr.com — 폰 브라우저에서 열고 "홈 화면에 추가"하면 오프라인 동작하는 설치형 앱이 됩니다.

## 주요 기능

- **4구 경로 제안** — 두께×당점 전수 시뮬레이션으로 경로 3개 추천, 몬테카를로 90회 시뮬레이션 예상 성공률·공 모임도(%) 표시, ▶ 재생으로 실제 궤적 애니메이션
- **뱅크샷 / 쿠션 수 규칙** — 빈쿠션 경로 탐색, 추천/1쿠션/2쿠션/3쿠션↑ 규칙 선택 (규칙까지 성공률 판정에 반영)
- **실력 3단계** — 초급(중심 반 팁 이내 당점·두꺼운 두께만·큰 실행 오차)/중급/상급
- **📷 사진 인식** — 경기면 모서리 자동 감지 → 호모그래피 원근 보정 → 카메라 위치 역산으로 쿠션 벽(37mm)·공(33mm) 높이 시차 보정 → 공 색 자동 분류. 전 과정 온디바이스(사진 전송 없음)
- **3구 시스템 계산기** — 파이브앤하프(코너-50)·플러스·볼 시스템 + 물리 경로 제안(쓰리쿠션 규칙). 첫 진입은 각 시스템의 **예시 배치**로 시작해 바로 성립하는 계산을 보여준다(사용자가 공을 놓는 순간부터는 그 배치를 유지)
- **경로별 추천 파워** — 표시할 경로마다 약·중·강 세 속도를 각각 24회씩 실측해 성공률이 가장 높은 속도를 고르고, 수구가 실제로 구른 거리를 레일 단위로 함께 표시
- **테이블 캘리브레이션** — 쿠션/구름 보정값을 기기에 저장해 내 구장에 맞춤 (4구·3구 두 탭 어디서나 조절 — 저장소가 하나라 양쪽이 함께 움직인다)

## 물리 엔진 (v4)

실측 비율(공 65.5mm / 대대 2844mm) 시간스텝 시뮬레이터 (DT 2.5ms):

- 미끄럼(μs=0.20)→구름(μr=0.012) 마찰 전이 — 끌기/밀기 곡선이 규칙이 아닌 물리로 발생
- 쿠션: 법선 반발 + 접선 마찰, 시네루 보정(회전 비율 연동 접선 상한 2/7→0.5, 스핀 홀드 0.25) — **파이브앤하프 차트로 캘리브레이션**
- 공-공: 반발 0.93 + 스로우 μ=0.06
- 검증 앵커: 알시아토레 실측(구름 1/2두께 분리각 33.7°, 스턴 탄젠트라인 등) + 파이브앤하프 교차검증 — `tests/validate-path.js`
- 알려진 한계: 깊은 뱅크(1쿠션수 30↑)에서 시뮬과 차트가 크게 갈린다(1쿠션 30 → 차트 20 / 시뮬 31). 교차 검증은 **1쿠션수 20~25 만 합격 판정**하고 30~35 는 정보로만 출력한다. 어느 쪽이 실제와 맞는지는 실측 재보정 전까지 알 수 없으므로, 앱은 그 구간의 파이브앤하프 결과에 **‘실험적’ 배지**를 붙인다(`expBadge()` · `tests/exp-badge-test.js`)
- 성공률은 실측이 아니라 시뮬레이션 예상치다. 재검사 24회에서 한 번도 재현되지 않거나 표시가 0%로 반올림되는 경로는 **제안하지 않는다** — 0% 경로를 '제안'이라 부를 수는 없다

## 프로젝트 구조

```
index.html           앱 전체 (단일 파일: 스타일 + 4구/3구 엔진 + 비전 + UI) — 루트에서 직접 서빙
privacy.html         개인정보처리방침 (수집 없음 / 사진 온디바이스)
terms.html           이용약관 (시뮬레이션 결과 면책)
manifest.webmanifest PWA 매니페스트 (start_url "./")
sw.js                서비스워커 (오프라인 캐시) — 배포 시 VERSION 올리기
vercel.json          보안 헤더(CSP 등) + 정적 캐시 + 옛 주소 308 리다이렉트
robots.txt           크롤러 정책 + sitemap 위치
sitemap.xml          색인 대상 URL (루트)
og-image.png         공유 미리보기 1200×630 — 재생성: python tools/gen-og.py
fonts/               자체 호스팅 폰트 (woff2 가변, latin 서브셋) — 외부 요청 제거
icons/               앱 아이콘 (tests 아님 — 재생성: 세션 스크립트 gen-icons.js)
tools/               개발용 스크립트 (배포 제외)
tests/               검증 스위트 21종 + 실환경 사진 픽스처(gzip)
.github/workflows/   CI — push/PR마다 tests/run-all.js
.vercelignore        tests/ tools/ .github/ README 를 배포에서 제외
```

**앱은 루트(`/`)에서 서빙된다** — 예전에는 `index.html` 이 `HelloBilli.html` 로 리다이렉트하는
껍데기였고 OG 태그를 양쪽에 중복 유지해야 했다. 지금은 앱 자체가 `index.html` 이라 페이지가 하나뿐이다.
옛 주소 `/HelloBilli.html` 은 `vercel.json` 의 `redirects` 로 308 처리된다 —
**이미 설치된 PWA 의 `start_url` 과 공유된 링크가 여기에 의존하므로 지우지 말 것.**

**외부 요청 0건** — 폰트를 자체 호스팅(`fonts/`)하므로 앱은 어떤 외부 도메인도 부르지 않는다.
CSP 가 `default-src 'self'` 로 잠겨 있으니, 외부 스크립트·글꼴·이미지를 추가하려면
`vercel.json` 의 CSP 를 함께 열어야 한다. (`sw.js` 는 자기 응답 헤더의 CSP 를 상속한다)

**폰트** — 둘 다 가변 폰트라 파일 하나가 모든 굵기를 담당한다. `unicode-range` 는 latin 서브셋이라
한글은 시스템 폰트로 폴백된다(두 폰트에 한글 글리프가 없어 기존과 동일). 갱신하려면
Google Fonts CSS 를 받아 latin `woff2` URL 을 뽑아 `fonts/` 에 덮어쓰고 `sw.js` VERSION 을 올린다.

## 테스트

```bash
node tests/run-all.js        # 전체 (~30초, 의존성 없음)
node tests/validate-path.js  # 물리 앵커만
```

- **엄격 테스트**(exit code): 물리 앵커, 사진 인식 합성 5시나리오, 실환경 필드 사진 2종
- **관대 테스트**(마커): 몬테카를로 분산이 있는 통합 테스트는 카드 개수 대신 단계 도달을 검증
- 물리 코드는 4구/3구 엔진이 2벌 존재하며 `validate-path.js`가 텍스트 동일성을 검사함 — **한쪽만 수정하면 CI가 실패**
  (비교 대상은 `stepBall`/`collideCushion` 두 함수뿐 — `solve3`·`phyParams` 는 3구 전용이라 자유롭게 고쳐도 된다)
- 3구 테스트는 배치·경로 좌표를 박아 두지 말 것 — `window.__hb3test`(`balls`/`play`/`sol()`)로 앱에서 읽는다.
  예시 배치나 캔버스 여백을 조정하면 하드코딩한 좌표는 조용히 의미를 잃고 테스트가 통과한 채로 아무것도 검사하지 않게 된다.
- 성공률이 몬테카를로라 `power-test`·`showall-test` 는 경로가 나올 때까지 재계산한다 (경로 수확량이 실행마다 흔들림)

## 배포

`main` push → Vercel 자동 배포. 큰 변경 시 `sw.js`의 `VERSION`(`hb-vN`)을 올려 구 캐시를 정리할 것.

**`vercel.json` 에 주석용 키를 넣지 말 것** — Vercel 이 스키마를 엄격히 검증해서
`_comment` 같은 임의 최상위 키가 있으면 **배포 자체가 실패한다**(`$schema` 는 예외로 허용).
설명은 이 README 에 적는다.
